# 第7問：ログイン処理のログ出力

**📅 出題日**: 2026年7月12日

## 📋 レビュー課題

ユーザーがメールアドレスとパスワードでログインする `LoginService` です。
`login()` を呼ぶと、メールとパスワードを照合して、成功なら `true` / 失敗なら `false` を返します。
あとから調査できるように、要所で `logger` を使ってログを残しています。

ローカルで正しいメール・パスワードを入れればログインでき、間違えれば弾かれ、
ログにもしっかり記録が残っていて、一見きちんと動いています。

対象: `bad/LoginService.java`

このコードを本番に出す前提で、「どこが問題か・どう直すべきか」をレビューしてみてください。
今回は特に **「ログに何を、どう出すべきか（ログ設計とセキュリティ）」** の観点で見てみましょう。

---

## ❌ NGポイント (bad/ 配下)

1. **平文パスワードをログに出力（最重要）**: `logger.info("... password=" + rawPassword)` や `error` ログで「入力パスワード」も「正しいパスワード」も出している。ログは運用チーム・監視SaaS・バックアップにまでコピーされ長期保存されるため、平文パスワードの漏洩箇所が爆発的に増える。さらに「正しいパスワード」がログに出せる＝**DBにパスワードを平文保存している**という二重の設計欠陥まで露呈している。

2. **メールアドレス（PII）を平文でログ出力**: メールは個人を特定できる個人情報（PII）。全ログに素で残すと、ログ流出＝個人情報流出になる。調査に必要ならマスキング（`ta***@example.com`）かユーザーIDのみに留める。

3. **業務エラーに `error` レベルを使用**: パスワードの打ち間違いは"ユーザーが日常的にやる操作（業務エラー）"であって"システム障害"ではない。`error` を使うと本物の障害ログが埋もれ、監視アラートがオオカミ少年化する。残すなら `info`/`warn`。

4. **User オブジェクトを丸ごとログ出力**: `logger.info("... =" + user)` で `toString()` が全フィールドを吐く。将来 User に住所・電話・クレカ番号が増えた瞬間、それらも自動でログに漏れる。出す項目は明示的に選ぶ（ID等）。

5. **ログの出し分けによるユーザー列挙（User Enumeration）**: 「ユーザーが存在しません」と「パスワード不一致」を区別できると、攻撃者がメールの登録有無を総当たりで判別でき、会員リストを作れてしまう。返却・ログとも存在有無を区別しない。

6. **`+` 文字列連結でのログ出力**: SLF4J のプレースホルダ `{}` を使うべき。(a) `+` は出力しない場合でも文字列化コストを常に払う（性能劣化）、(b) 外部入力(email)をそのまま連結すると改行を仕込まれ偽ログを捏造される（ログインジェクション）。

## ⭕️ 改善案 (good/ 配下)

1. パスワードは一切ログに出さない。照合は `PasswordEncoder.matches()` で行い、DBにはハッシュ化した値を保存する（平文を変数やログに"寄り道"させない）。
2. メールは `maskEmail()` でマスキングしてから出力する。
3. ログ出力は `{}` プレースホルダに統一する。
4. ログイン失敗は「ユーザー不在」「パスワード違い」を区別せず、同一の `info` ログ＋同一の戻り値で扱う（列挙対策）。成功時も `userId` だけを出す。
5. 呼び出し側のユーザー向けメッセージも「メールアドレスまたはパスワードが正しくありません」に統一する。

## 💬 レビューコメント例文

> LoginService、動作は問題ないです。仕上げにログまわりだけ一緒に固めさせてください。
> まず `password` がログに出てしまっているので、ここは必ず消しましょう。ログって開発者だけのものに見えて、実は運用・監視・バックアップまでコピーされて長く残るんです。「正しいパスワード」まで出ているのは、裏でパスワードを平文保存している可能性もあるので、そこも合わせて `PasswordEncoder` でのハッシュ照合に寄せたいです。
> あと `email` は個人情報なので `ta***@...` のようにマスキングを。パスワード違いは"ユーザーの打ち間違い"＝業務エラーなので、`error` だと本物の障害が埋もれます。`info` に落としましょう。
> 細かいですが「ユーザーがいない/パスワード違い」の出し分けは、攻撃者に会員かどうかを教えてしまう（ユーザー列挙）ので、失敗は一律同じ扱いにするのが安全です。ログは `+` 連結より `{}` の方が性能的にもインジェクション対策的にも安心なので、そこも揃えておきましょう。
