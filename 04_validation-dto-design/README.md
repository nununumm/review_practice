# 第4問：ユーザープロフィール更新API（入力チェックとデータの受け渡し）

**📅 出題日**: 2026年7月10日

## 📋 レビュー課題

ユーザーの「名前・メールアドレス」を画面から変更できるプロフィール更新APIです。
`GET /users/{id}` で現在の情報を取得し、`PUT /users/{id}` で送られてきた内容に上書き保存します。
Postman で叩くと期待通りに名前やメールが更新され、一見きちんと動いています。

対象: `bad/ProfileController.java`, `bad/User.java`

このコードを本番に出す前提で、「どこが問題か・どう直すべきか」をレビューしてみてください。

---

## ❌ NGポイント (bad/ 配下)

1. **マスアサインメント脆弱性（受け取りすぎ）** 🔑: 更新APIが `@RequestBody User` でエンティティを丸ごと受け取り、`setRole()` / `setPassword()` まで呼んでいる。建前は「名前・メール編集」だが、リクエストに `"role":"ADMIN"` を紛れ込ませれば **一般ユーザーが自力で管理者に昇格できる**。触らせてはいけない項目に書き換え口を開けている。
2. **情報漏洩（返しすぎ）** 🔒: `GET` がエンティティをそのまま返すため、レスポンスJSONに `password`（ハッシュ済みでも）と `role` が含まれてしまう。出す必要ゼロの内部情報が通信・キャッシュ・フロントに流出する。
3. **入力バリデーションの欠如**: `email` が空文字や `"hello"`（＠なし）でも素通りして保存される。必須チェック・形式チェックが一切ない。
4. **エンティティとAPIの癒着（DTO不在）**: DBの形（Entity）を入口・出口に直結しているため、DBカラムを足すと勝手にAPI仕様が変わる。APIだけの都合（項目名変更・算出項目の追加）にも対応できない。
5. **存在チェックなし**: `findById(id).get()` は対象が無いと `NoSuchElementException` で500エラー。
6. **Fat Controller**: 「取得→各項目を上書き→保存」という業務ロジックが Controller に直書きされ、テストしづらく再利用も効かない。

## ⭕️ 改善案 (good/ 配下)

1. **用途別のDTOを用意する**: 受信用 `UserUpdateRequest`（`name`/`email` **だけ**）と返信用 `UserResponse`（`id`/`name`/`email` **だけ**）を分ける。これだけで **1（受け取りすぎ）と 2（返しすぎ）が構造的に消える**。「箱に入り口が無ければ、そもそも入れられない」。
2. **バリデーションを宣言する**: 受信DTOに `@NotBlank` / `@Email` を付け、Controllerの引数を `@Validated`（または `@Valid`）で起動する。違反時は `@RestControllerAdvice` で `400 Bad Request` を返す。
3. **存在チェックは `orElseThrow(...)`**: 無ければ意味のある例外（→ `404 Not Found`）にする。
4. **ロジックはServiceへ**: Controllerは「受け取ってDTOで返すだけ」に薄くし、更新処理は `UserService` に委譲する。
5. **エンティティ↔DTOの変換を1箇所に集約**: 詰め替え処理を明示的に書き、外に出す項目・受け取る項目を意図的にコントロールする。

## 💬 レビューコメント例文

> プロフィール更新API、動作はバッチリでした！本番向けに、セキュリティ観点で2点だけ一緒に固めさせてください。
>
> 1点目がいちばん大事で、**更新APIが `User` エンティティを丸ごと受け取っている**ところです。今の作りだと、リクエストに `"role":"ADMIN"` を混ぜて送られると、一般ユーザーが自分を管理者に昇格できてしまいます（マスアサインメントと呼ばれる典型的な穴です）。**「名前とメールだけ」を持つ更新用DTOを1枚挟む**と、そもそも `role` を受け取る口が無くなるので、この穴が構造ごと塞げます。
>
> 2点目は、`GET` が `User` をそのまま返していて、レスポンスに `password` と `role` まで出ている点です。こちらも**返信用DTO（id/name/emailだけ）**に詰め替えて返す形にしませんか？
>
> ついでに、受信DTOに `@NotBlank` / `@Email` を付けておくと空メールや不正な形式も弾けます。存在チェックの `.get()` も、前回の要領で `orElseThrow` にできると綺麗です。DTOの導入は最初こそ手間に見えますが、**「DBの形」と「APIの形」を切り離す**とこの手の事故がまとめて防げるので、投資に見合いますよ。次回ペアで変換まわり一緒に見ましょう！
