## 復習履歴

### 2026-07-19 ｜ 第4問 プロフィール更新API（DTO設計・バリデーション）

形式：`bad/` のみを見て、記憶で問題点を指摘 → 当時の模範解答と答え合わせ。

**自力で挙げられた点**
- Controller にロジックをベタ書き → サービス層へ切り出す（責務分離）
- Entity をそのままレスポンスに返すと password / role が漏れる → レスポンス用 DTO で絞る（情報漏洩）
- `findById(id).get()` の存在チェック漏れ

**ヒントを経て到達した点**
- **マスアサインメント脆弱性**：`@RequestBody User` で受けると、クライアントが `role` を送り込んで自分を管理者に昇格できる。→ 入力用 DTO（`name` / `email` だけ）で「そもそも受け取らない」構造にする。
  - 当初は「認証を挟んで守る」止まり → 「入り口の型で構造的に受け取らない」に発想を更新。
- **バリデーション**：`@Valid`（Controller 引数）＋ DTO 側の `@NotBlank` / `@Email` はセットで機能する。

**本人が整理できた原則**
- Entity は「DTO の一種」ではなく別物。Entity ＝ 名簿（社外秘）、DTO ＝ 名刺（用途別に渡してよい情報だけ）。
- 「入力用 / 出力用 / Entity」の3つは初学者の標準構成だが固定ではない。原則は
  **「Entity は絶対に外に出さない。窓口には用途に必要な項目だけの箱（DTO）を立てる。形が同じなら使い回し、違うなら分ける」**。

**総評 / 伸びしろ**
- セキュリティ・アーキテクチャの「気づく力」は合格ライン。
- 伸びしろは「守り方の引き出し」。「認証を挟む」より先に「入り口の DTO で構造的に受け取らない」を第一候補に。
