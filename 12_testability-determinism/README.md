# 第12問：パスワード再設定の申請サービス（テストのしやすさ）

**📅 出題日**: 2026年7月19日

## 📋 レビュー課題

ユーザーが「パスワードを忘れた」ときに、再設定を申請する `PasswordResetService` です。
`requestReset(email)` を呼ぶと、①再設定用のトークン（合言葉）を発行し、
②有効期限（1時間）を付けて保存し、③そのリンクをメールで送る、という処理をします。

ローカルで動かすと、トークン付きのメールがちゃんと飛んできて、一見きちんと動いています。

対象: `bad/PasswordResetService.java`

このコードを本番に出す前提で、「どこが問題か・どう直すべきか」をレビューしてみてください。
今回は特に **「このサービスをどうやって自動テストで守るか（テスト容易性）」** の観点で見てみましょう。

> ヒント：「発行されたトークンが正しく保存されるか」「有効期限は本当に1時間後か」「ちゃんとメールを送ろうとするか」を
> あなたが JUnit で確かめようとしたら、素直にテストが書けるでしょうか？書きにくいなら、それは**設計のにおい**です。

---

## ❌ NGポイント (bad/ 配下)

1. **メール送信オブジェクトを `new` で内部生成している（最重要）**: `new SmtpMailSender("smtp.example.com", 587)` をメソッド内で直接作っているため、テストのたびに本物のSMTPサーバーへ実際にメールを送りにいってしまう。しかも `new` で固く結びついているので、「送ったフリ（モック）」に差し替える隙間が無い。＝**自動テストで守れない。**
2. **`System.currentTimeMillis()` への直接依存（＝非決定的）**: 「今、何時か」をコードに直接焼き込んでいるため、時刻を固定できない。「有効期限は本当に“今＋1時間”か？」を検証しようにも、コードが動いた瞬間の「今」とテストが照合する「今」が必ずズレて、ピッタリの検証ができない。
3. **マジックナンバー**: `60 * 60 * 1000`（有効期限）や `"smtp.example.com"` / `587`（接続先）がコードに直書きされ、意味も分かりにくく、変更・テストがしづらい。
4. **`@Autowired` によるフィールド注入**: 依存が外から見えず、テスト時にモックを差し込むのに苦労する（リフレクション頼みになる）。「このクラスが何を必要とするか」がコンストラクタに現れない。
5. **`UUID.randomUUID()` の直接生成**: 発行トークンが毎回変わるため、「保存された値」と「メール本文の値」の一致をテストで照合しづらい。

## ⭕️ 改善案 (good/ 配下)

1. **メール送信を `MailSender` インターフェースにして注入**: 本番は `SmtpMailSender`、テストはモックを差し込む。実際に送らずに「send が呼ばれたか・宛先と本文は正しいか」を検証できる。
2. **`Clock` をコンストラクタで注入**: `System.currentTimeMillis()` をやめ、時刻は `Clock` から取得。テストでは `Clock.fixed(...)` で「今」を固定でき、有効期限を**ピッタリ**検証できる。有効期限の“長さ”は `PasswordResetProperties`（`Duration`）に外部化。
3. **設定値を `PasswordResetProperties` に外部化**: 有効期限やSMTP設定を `application.yml` から注入。マジックナンバーを排除し、テストでは短い期限に差し替え可能。
4. **コンストラクタ注入 + `final`**: 必要な部品が一目で分かり、テストでは `new` で普通に組み立てられる。
5. **`TokenGenerator` を注入**: トークン発行を差し替え可能にし、テストで固定値にできる。
6. **有効期限の計算はJava側（Clock）に保つ**: 「隙間」を気にして時刻計算をSQL(`@Query`/`NOW()`)へ寄せると、DB時刻を固定できずテストできなくなる。独自SQLが活きるのは「期限切れトークンの一括削除」など別用途（`good/PasswordResetTokenRepository.java` 参照）。

> 💡 2〜5の共通の考え方は **「変わるもの・外部に触れるものは、自分で作らず“外から渡す”（DI）」**。
> これ一つで「メールの `new`」も「時刻の直接依存」もまとめて解決できる。

## 💬 レビューコメント例文

> パスワード再設定、機能はきちんと動いていますね！ここからは「本番で自動テストに守らせる」観点で2つ相談させてください。
>
> 1つ目、メール送信の `new SmtpMailSender(...)` の部分です。いまのままだとテストのたびに本物のメールが飛んでしまい、CIで安全に回せません。`MailSender` をインターフェースにしてコンストラクタで受け取る形にすると、テストではモックに差し替えて「送ろうとしたか」を検証できるようになります。
>
> 2つ目、有効期限の `System.currentTimeMillis()` です。ここが直接依存だと「本当に1時間後か？」を固定して確かめられないんです。`Clock` を注入して `clock.instant()` から求める形にすると、テストで時刻を固定できて期限をピッタリ検証できます。有効期限の“長さ”自体も設定クラスに出しておくと、テストで短くできて便利です。
>
> どちらも「変わるもの・外に触れるものは自分で `new` せず外から渡す」という同じ発想です。良ければ一緒に直しましょう！
