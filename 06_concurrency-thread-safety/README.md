# 第6問：請求書番号を発行するサービス（同時アクセス）

**📅 出題日**: 2026年7月11日

## 📋 レビュー課題

請求書番号（例: `INV-20260711-0001`）を発行する Spring Boot のサービスです。
`issue()` を呼ぶたびに「今日の日付＋その日の連番」で番号を1つ作って返します。
1人で順番に使うぶんには、ちゃんと連番が振られて動いているように見えます。

対象: `bad/InvoiceNumberService.java`

このサービスは Web アプリの一部で、**多くの利用者が同時に請求書を発行する**場面で使われます。
「どこが問題か、どう直すべきか」をレビューしてみてください。

---

## ❌ NGポイント (bad/ 配下)

1. **連番の競合（レースコンディション）＝ 最重要**:
   `@Service` は「アプリに1個だけ」作られて全リクエストで共有される（＝シングルトン）。
   その共有フィールド `sequence` を `sequence = sequence + 1` で増やしているが、
   これは「①読む → ②+1する → ③書き戻す」の3ステップに分かれる。
   複数リクエストが同時に実行すると、2人が同じ値を読んで**同じ番号を発行**してしまう。
   請求書番号のダブりは経理・監査に直結する重大事故。

2. **`SimpleDateFormat` はスレッドセーフでない**:
   `SimpleDateFormat` は内部で状態を持ち回す作りで、複数スレッドで同時に使うと
   日付文字列が壊れたり例外が飛ぶ。これを共有フィールドに置いているのは典型的なアンチパターン。

3. **`lastIssuedNumber` を共有フィールドで持っている**:
   全リクエストで1個を共有するため、Aさんの発行直後にBさんの発行で上書きされ、
   `getLastIssuedNumber()` が他人の番号を返す。リクエストごとの状態をサービスに持たせてはいけない。

4. **連番をメモリで管理している（そもそもの設計）**:
   メモリ上の連番は「再起動で消える（→番号が巻き戻る）」「サーバーを複数台に増やすと
   各台がバラバラの連番を持つ（→台をまたいでダブる）」という現場のリアルに耐えられない。

## ⭕️ 改善案 (good/ 配下)

1. **連番の置き場所をメモリ → DB に移す**:
   「日付ごとの連番」を専用テーブル（`InvoiceSequence`）で管理する。
   DBは再起動で消えず、全サーバーで1つを共有できる。

2. **DBの行ロックで1人ずつ処理する**:
   `@Lock(PESSIMISTIC_WRITE)` で連番の行に鍵をかけ、`@Transactional` の間だけ他を待たせる。
   「A案（鍵をかける）」と「B案（自前メモリで持たない）」を合体させた形。

3. **`SimpleDateFormat` → `DateTimeFormatter`**:
   スレッドセーフなので `static final` で1個を共有してよい。

4. **リクエスト固有の状態（直近番号）をサービスに持たせない**:
   `lastIssuedNumber` フィールドと `getLastIssuedNumber()` は削除。必要なら呼び出し側で保持する。

## 💬 レビューコメント例文

> このサービス、1人で使うぶんには完璧に動くのですが、本番で同時アクセスされる前提で見てみましょう。
> `@Service` はアプリ内で1個だけ共有されるので、`sequence` も全リクエスト共通の箱になります。
> `sequence = sequence + 1` は「読む→足す→書く」の3手に分かれるため、
> 同時に来たリクエスト同士が同じ番号を発行してしまう競合（レースコンディション）が起きえます。
> 請求書番号のダブりは経理に直結するので、ここは連番を DB のテーブルで管理して、
> 行ロック（`@Lock(PESSIMISTIC_WRITE)`）＋ `@Transactional` で1件ずつ採番する形にしませんか？
> ついでに `SimpleDateFormat` もスレッドセーフでないので `DateTimeFormatter` に替えておくと安心です。
> `lastIssuedNumber` の共有フィールドも他人の番号が返る原因になるので、削除でよさそうです。

### 📌 このお題で学んだこと（観点：並行処理 / スレッドセーフ）
- `@Service` はシングルトン。**サービスに可変フィールドを持たせない**のが原則。
- 「読む→変更→書く」の複合操作は、割り込みで壊れる（＝アトミックでない）。
- ロックには「アプリ内のロック（`synchronized`/`Atomic系`）」と「DBの行ロック」があり、
  **複数台・再起動に耐える必要があるならDB側**で守るのが定石。
