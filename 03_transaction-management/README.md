# 第3問：注文処理サービス（在庫・注文・ポイントの一括更新）

**📅 出題日**: 2026年7月10日

## 📋 レビュー課題

ECサイトの注文処理を担う `OrderService` です。
1件の注文で「①在庫を減らす → ②注文を保存する → ③ユーザーにポイントを付与する」という
3つのDB更新をまとめて行っています。単体テストでは正常に動いているように見えます。

対象: `bad/OrderService.java`

このコードを本番に出す前提で、「どこが問題か・どう直すべきか」をレビューしてみてください。

---

## ❌ NGポイント (bad/ 配下)

1. **トランザクション管理の欠如（部分コミット）**: `@Transactional` が無く、`save()` ごとに確定してしまう。途中の②や③で失敗すると「在庫だけ減って注文が消える」等の不整合が残る。
2. **例外の握りつぶし（ロールバックを殺す）**: `try-catch` で例外を飲み込み正常 return しているため、`@Transactional` を付けても Spring のプロキシからは「成功」に見え、**ロールバックが効かない**。`printStackTrace()` はログ基盤に乗らず追跡不能。
3. **存在チェックなし**: `findById(...).get()` は対象が無いと `NoSuchElementException`。`findByUserId(...)` は戻り値 null で NPE のリスク。
4. **業務バリデーションの欠如**: 在庫超過や注文数0・マイナスを弾かず、在庫がマイナス値でも平然と保存される。
5. **同時実行制御なし（ロストアップデート）**: 「在庫を読む→計算→書く」の隙間で、同時注文により在庫がマイナスになりうる。
6. **APIレスポンス設計の弱さ**: 成否を文字列で返しており、HTTPステータスや構造化エラーになっていない。

## ⭕️ 改善案 (good/ 配下)

1. Service メソッドに **`@Transactional`** を付け、3更新を1トランザクションに束ねる。
2. Service では**例外を握りつぶさず投げる**。エラーの見せ方は **`@RestControllerAdvice`（GlobalExceptionHandler）** に一元化し責務を分離。ログは **SLF4J ロガー**で記録。
3. `Optional` は `orElseThrow(...)` で存在を明示的に扱う。
4. 業務バリデーション（注文数 >= 1 等）を実装。
5. 在庫減算を **条件付きの原子的 UPDATE**（`WHERE quantity >= :count`）にし、更新0件を在庫不足として扱う。1文で読み取り＋判定＋書き込みが完結し、同時実行でもマイナスにならない。（代替案: 楽観ロック `@Version` / 悲観ロック `@Lock`）
6. Controller は薄く保ち、`201 Created` と DTO を返す。

## 💬 レビューコメント例文

> 注文処理お疲れさまです！ロジックの流れはとても分かりやすかったです。本番に出す前に、2点だけ一緒に固めさせてください。
>
> 1つ目は**トランザクションです**。今は在庫・注文・ポイントの3更新が別々に確定するので、途中で失敗すると「在庫だけ減る」不整合が残ります。メソッドに `@Transactional` を付けて1まとまりにしましょう。
>
> 2つ目が少しトリッキーなのですが、**今の `try-catch` があると、その `@Transactional` を付けてもロールバックが効きません**。Spring は「例外がメソッド外へ飛び出したか」で巻き戻しを判断するので、ここで握りつぶすと“成功扱い”でコミットされてしまうんです。エラーの見せ方は `@RestControllerAdvice` に寄せて、Service は例外をそのまま投げる形にしませんか？
>
> あと在庫は、同時注文でマイナスにならないよう `WHERE quantity >= :count` の条件付き UPDATE にできるとより堅牢です。ここは少し高度なので、よければ次回ペアで見ましょう！
