# 第5問：商品取得・購入APIのエラー処理

**📅 出題日**: 2026年7月11日

## 📋 レビュー課題

オンラインショップの商品を扱う `ProductController` です。

- `GET /api/products/{id}` … 商品を1件取得する
- `POST /api/products/{id}/purchase?quantity=2` … 商品を購入して在庫を減らす

ローカルで正常なIDを叩くとちゃんと商品が返り、購入もでき、一見きちんと動いています。

対象: `bad/ProductController.java`

このコードを本番に出す前提で、「どこが問題か・どう直すべきか」をレビューしてみてください。
特に **「想定外のことが起きたとき（＝異常系）にどう振る舞うか」** の観点で見てみましょう。

---

## ❌ NGポイント (bad/ 配下)

1. **`Optional.get()` の乱用**: `findById(id).get()` は「中身が必ず入っている前提」で取り出す乱暴なメソッド。存在しないIDだと `NoSuchElementException` が飛ぶ。存在チェックをサボっている。
2. **例外の握りつぶし＋`return null`（getProduct）**: `catch` で例外を飲み込み `return null` するため、Springからは「成功(200)して中身null」に見える。フロントは「商品が無い」のか「本当にnull」なのか判別できず、別の場所でぬるぽ落ちする。原因追跡が地獄。
3. **異常時のHTTPステータスがバラバラ**: 同じ「存在しないID」でも、`getProduct` は `200＋null`、`purchase` は `500`。同じシステムで挙動が不統一で信頼できない。本来は `404` に揃えるべき。
4. **グローバル例外ハンドラの不在**: `@RestControllerAdvice` が無く、エラーを正しいステータスコードに変換する仕組みが丸ごと欠けている。
5. **業務エラーに雑な `RuntimeException`（purchase）**: 「在庫不足」はバグではなく想定内の業務状況。専用の例外型で表現すべき。また `500` になってしまい「利用者のせい」を「サーバーのせい」にしている（本来 `409`）。
6. **Fat Controller**: 在庫確認・減算・保存という業務ロジックが Controller に直書き。サービス層に分離すべき（テスト容易性）。
7. **ログが闇に消える**: `System.out.println` / `printStackTrace()` は本番のログ基盤に乗らず、障害を検知・追跡できない。ロガー（SLF4J）を使うべき。
8. **エラーメッセージの情報漏洩**: 在庫不足時に現在の在庫数をレスポンスに載せている。内部情報を出しすぎ。

## ⭕️ 改善案 (good/ 配下)

1. **`.get()` → `orElseThrow(...)`**: 「無ければ専用例外を投げる」を明示（`ProductNotFoundException`）。
2. **専用の例外クラスを定義**: `ProductNotFoundException` / `OutOfStockException` を作り、「エラーの種類」を型で表現する。
3. **`@RestControllerAdvice` に裁きを集約**: Controller/Service は「例外を投げるだけ」、裁くのは受付係1箇所。例外の種類ごとに `404 / 409 / 400 / 500` を割り当てる。
4. **500は最後の網＋ログ**: `@ExceptionHandler(Exception.class)` で想定外を受け止め、`log.error(..., e)` でスタックトレースごと記録。利用者には定型文のみ返す。
5. **統一エラーレスポンス（`ErrorResponse`）**: エラーは必ず `{status, message}` の同じ形のJSONで返す。
6. **サービス層への分離＋`@Transactional`**: 業務ロジックを `ProductService` に移し、購入処理はトランザクションで保護。
7. **入力バリデーション**: `@Min(1)` で `quantity` を検証し、不正入力は `400` に。

### 設計の勘所（今回の学び）
- HTTPステータスの軸：**4xx = クライアントのせい（想定して先回りで拾う） / 5xx = サーバーのせい（想定外を網で受け止めログに残す）**。
- 各Controllerで try-catch して一つずつ裁くのはアンチパターン。**「投げる場所」と「裁く場所」を分離**する。
- 500を無くすのが目的ではなく、**「500に落ちるのは本物の想定外だけ」という状態にする**のが目的。

## 💬 レビューコメント例文

> `getProduct` の `findById(id).get()` ＋ `catch → return null` の流れ、ここだけ相談させてください🙏 存在しないIDのとき、いまの実装だと「200 OKで中身null」が返ってしまい、フロント側で『商品が無い』のか『本当にnull』なのか区別できないんですよね。かつ `purchase` の方は同じ状況で500になっていて、挙動が揃っていないのも気になりました。
>
> 提案としては、①`orElseThrow(() -> new ProductNotFoundException(id))` で「無ければ専用例外を投げる」に統一し、②`@RestControllerAdvice`（グローバル例外ハンドラ）を1つ用意して、そこで `ProductNotFoundException → 404`、在庫不足 → `409` のように種類ごとにステータスを割り当てる形にしませんか？ こうすると各Controllerからtry-catchが消えて業務ロジックがスッキリしますし、エラー時のレスポンス形式もチームで統一できます。ログも `printStackTrace` だと本番の基盤に乗らないので、SLF4Jのロガーに寄せておけると運用時に助かります💪
