# 学習ノート：例外処理・エラーレスポンス設計の考え方

> 第5問の学習中に交わした議論のまとめ。復習用。

---

## 1. HTTPステータスコードの基本の軸

- **4xx = クライアント（利用者）のせい** … リクエストがおかしい、無いものを要求した
- **5xx = サーバー（こちら）のせい** … サーバーが壊れた

| ケース | あるべき番号 | 考え方 |
|---|---|---|
| 存在しないID | **404 Not Found** | 探し物が無い |
| 在庫不足で買えない | **409 Conflict**（or 400） | リクエストは正しいが、データの状態と衝突して実行不可 |
| quantityにマイナス等の変な値 | **400 Bad Request** | 入力の形式がそもそもおかしい |
| DBダウン等の本当の想定外 | **500 Internal Server Error** | サーバー側の障害。500は本来これだけに使う |

- 400 vs 409 の覚え方：**入力の形式がおかしい → 400 / データの状態とぶつかる → 409**。

---

## 2. なぜ「200になってしまう」のか

- ステータスは **「メソッドが例外を投げずに普通に return できたか」** で決まる。
  - 普通に return → Spring は「成功」とみなす → **200 OK**
  - 例外がメソッドの外まで飛び出す（誰もcatchしない）→ **500**
- `catch → return null` は「握りつぶして普通にreturnした」ので、**エラーなのに200＋nullになる**。これが罠。
- try-catchが原因で200になるのではなく、**「catch後に握りつぶして普通にreturnした」から200に見える**、が正確な理解。

---

## 3. 正しい設計：「投げる場所」と「裁く場所」を分ける

- **アンチパターン**：各Controllerメソッドで try-catch して一つずつ裁く → 全メソッドに同じtry-catchがコピペされ、業務ロジックが埋もれる。
- **正しい型**：
  - Controller / Service は、エラーだと思ったら **「例外を投げるだけ」**（try-catchしない）。
  - 裁くのは **`@RestControllerAdvice`（グローバル例外ハンドラ）1箇所に集約**。例外の種類ごとに番号を割り当てる。
- たとえ話：各社員（API）がトラブルを自席で全部対応せず、**1階の受付（受付係）に全部回す**。受付が種類ごとに正しく捌く。

### 例外クラスは別途作る必要がある
- `ProductNotFoundException` などの例外クラスは **独立した1ファイルとして自分で作る**。
  - 中身は「`RuntimeException` を継承して名前を付けただけの小さな箱」でOK。
  - なぜ名前を付ける？ → ただの `RuntimeException` だと受付係が「これは404にすべき」と**見分けられない**から。
- `GlobalExceptionHandler` は **エラーを「作る」場所ではなく「裁く」場所**。「この例外型 → この番号」と対応づけを書くだけ。

| ファイル | 役割 |
|---|---|
| `XxxException.java` | エラーの「種類」を定義する（箱を作る） |
| `GlobalExceptionHandler` | その種類を「何番で返すか」割り振る（1箇所に集約） |

---

## 4. 500は「消す」のではなく「網」として残す

- 開発時に全エラーを予見しきることはできない（人間だから漏れる）。
- だから `@ExceptionHandler(Exception.class)` を **最後のセーフティネット**として必ず1つ残す。
  - 想定外を漏らさず受け止め、**500のときだけ error レベルでスタックトレースごとログに残す**（本物の障害＝調査が必要だから）。
  - 利用者には内部詳細を見せず、定型文だけ返す（情報漏洩防止）。
- **目的は「500を無くすこと」ではなく「500に落ちるのは本物の想定外だけ、という状態にすること」**。

### 認識の補強：4xxは「先回りで拾う」
- ❌ 開発中にたまたま踏んだエラーをその都度4xxにする（テストで踏まないと本番で500漏れ）
- ⭕️ 設計段階で「利用者はどう間違える？」を**先回りで洗い出し**、4xxに割り振る

---

## 5. システムの異常は4種類

| 種類 | 例 | 「設計漏れ」か | 対応 |
|---|---|---|---|
| ① 想定内のエラー | 存在しないID、在庫不足、入力ミス | 拾うのが設計 | 4xxに割り振る |
| ② 想定漏れのエラー | nullを考慮してなくてNPE | ✅ まさに設計漏れ | 気づいたら①に昇格 |
| ③ ロジックのバグ | 計算式・条件分岐ミス | 実装ミス | コードを直す（500で検知） |
| ④ 環境・インフラ障害 | DBダウン、通信タイムアウト | 漏れではなく前提 | 500＋ログ、耐障害設計 |

- インテグレーションテストで見つかるバグ = 主に **②（想定漏れ）**。単体テストでは正常系しか見ないので、繋げて動かすと考慮漏れが出る。
- ただし③（実装ミス）や④（インフラ障害）は「エラーの割り振り漏れ」とは別物。
- **健全なサイクル**：①で拾えるだけ拾う → 予見しきれない②③④を500の網で受ける → ログで学んで①に昇格させ直す。これを繰り返してシステムが堅牢になる。
