# 第9問：注文APIのコントローラ（REST API 設計）

**📅 出題日**: 2026年7月18日

## 📋 レビュー課題

ECサイトの「注文（Order）」を扱う REST コントローラです。
注文の一覧取得・1件取得・作成・削除ができるように作られています。

ローカルで叩くと、それぞれ期待した動き（一覧が返る、作れる、消せる）をするので、
一見きちんと動いているように見えます。

対象: `bad/OrderController.java`

このコードを「外部にも公開する Web API」として本番に出す前提で、
「どこが問題か・どう直すべきか」をレビューしてみてください。
今回は特に **「Web API としての設計作法（REST の観点）」** に注目してみましょう。

---

## ❌ NGポイント (bad/ 配下)

1. **【最重要】HTTPメソッドを限定していない（`@RequestMapping` だけ）＝ GETで削除・作成ができてしまう**: `@RequestMapping("/deleteOrder")` はメソッド指定がないため GET でも呼べる。ブラウザのアドレスバーに `/deleteOrder?id=1` と打つだけで注文が消せる状態。検索エンジンのクローラーがリンクを辿っただけでデータが消える、CSRF（＝罠サイト経由で勝手に操作させる攻撃）に弱い、といった重大事故につながる。「見るだけ(GET)」と「変更する(POST/DELETE)」は必ず分ける。
2. **ステータスコードが常に200 OK**: 注文が見つからないのに「空のOrder」を200で返し、作成失敗も `"error: ..."` を200で返す。呼び出し側は成否を判定できず、文字列を目で読むしかない。正しくは 404 Not Found / 400 Bad Request / 201 Created などをステータスコードで伝える。
3. **URLが動詞ベース（`/getAllOrders` `/createOrder` `/deleteOrder`）**: RESTでは「URL＝モノ(名詞)」「HTTPメソッド＝動作(動詞)」。URLに動詞を入れると動作の数だけURLが増えて散らかる。`/orders` に統一し、動作はメソッドで表す。
4. **Entityを直接返している（`List<Order>` / `Order`）**: DB直結クラスをそのまま返すと、見せたくない項目（原価・内部フラグ・他人の情報）までJSONに漏れる。返却専用のDTOに詰め替える。
5. **一覧が全件取得（`findAll`）**: 件数無制限。データが増えるとメモリを食い潰し落ちる。ページング（区切って返す）を入れる。
6. **成否を文字列で返す（`"success"` / `"deleted"` / `"error: ..."`）**: 機械から扱いにくい。成否はステータスコードで、データはJSONで返す。削除成功は 204 No Content が定石。
7. **作成の try/catch で例外を握りつぶし**: 例外の中身を文字列にして200で返している。例外は投げ、共通の `@RestControllerAdvice` で適切なステータスに変換するのが筋。
8. **入力チェックが無い**: `quantity` にマイナスや0が来ても素通り。`@Valid` ＋ DTOのバリデーションで入り口で弾く。
9. **フィールドインジェクション（`@Autowired private`）**: コンストラクタ注入にすると、依存が明示され・テストしやすく・`final` で不変にできる（第8問の学び）。

## ⭕️ 改善案 (good/ 配下)

1. **URLは名詞に統一し、`@GetMapping`/`@PostMapping`/`@DeleteMapping` でメソッドを限定**（`/orders`, `/orders/{id}`）。GETで作成・削除できる事故を根絶する。
2. **ステータスコードで語る**: 取得成功=200、作成成功=201 Created（+Locationヘッダ）、削除成功=204 No Content、未存在=404、入力不正=400。
3. **返却はレスポンスDTO（`OrderResponse`）に詰め替え**、Entityを外に出さない。
4. **一覧は `Pageable` でページング**し、全件取得をやめる。
5. **作成は `@RequestBody` + `@Valid` でJSON本文を受け、入力を検証**。IDや数量をURLにぶら下げない。
6. **例外は投げて `@RestControllerAdvice` に集約**。Controllerから try/catch を消し、本来の処理に集中させる。
7. **コンストラクタ注入**に変更。

## 💬 レビューコメント例文

> `deleteOrder` の件、実装ありがとうございます! 動作はOKでした。1点だけ本番前に直したいところがあって、いまアノテーションが `@RequestMapping` なので **GETでも削除が呼べてしまう**状態なんです。そうすると検索クローラーがURLを辿っただけで注文が消える、といった事故が起きえます。ここは `@DeleteMapping("/{id}")` に変えて「消す操作はDELETEでしか呼べない」ように締めておきましょう。ついでにURLも `/orders/{id}` の形にすると、他のAPIとも作法が揃ってきれいになります 👍

> レスポンスまわり、`Order` をそのまま返しているのが少し気になりました。エンティティを直接返すと、今後カラムが増えたときに**見せたくない項目まで自動でJSONに載ってしまう**リスクがあります。`OrderResponse` みたいな返却専用クラスを1枚かませて、「外に出す項目」を明示的にコントロールしておくと安全です。あと、見つからなかったときは空オブジェクトではなく **404を返す**と、呼び出し側が成否を判定しやすくなりますよ。
