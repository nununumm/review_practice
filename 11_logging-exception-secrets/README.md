# 第11問：外部決済ゲートウェイ呼び出しサービス（ログと例外の扱い）

**📅 出題日**: 2026年7月19日

## 📋 レビュー課題

外部の決済サービス（カード決済のゲートウェイ）に、カード決済を依頼する `PaymentGatewayService` です。
`charge()` を呼ぶと決済を実行し、成功なら `true` / 失敗なら `false` を返します。
あとから決済トラブルを調査できるように、開始時・レスポンス時・失敗時にログを残しています。

ローカルでテスト用カードを使うと決済が通り、失敗時にも `false` が返り、
ログにもしっかり記録が残っていて、一見きちんと動いています。

対象: `bad/PaymentGatewayService.java`

このコードを本番に出す前提で、「どこが問題か・どう直すべきか」をレビューしてみてください。
今回は特に **「ログに何をどう残すか」と「例外（失敗）をどう扱うか」** の観点で見てみましょう。

---

## ❌ NGポイント (bad/ 配下)

### 【ログ編】

1. **秘密情報・個人情報をログに垂れ流している（最重要）**: 21〜24行目で、カード番号・CVV・APIトークンを平文でログに出力している。ログは運用チームや外部のログ集約基盤など多くの目に触れ、ファイルに残り続ける。ここに秘密情報が載ると情報漏洩に直結し、**PCI-DSS（カード情報取り扱いの世界共通ルール）違反**になる。
2. **CVV（セキュリティコード）は、そもそも保存・出力自体が禁止**: カード番号以上に厳しい扱い。PCI-DSS では認証後の CVV の保存を明確に禁じている。「マスクすればOK」ですらなく、ログにもDBにも一切残してはいけない。
3. **レスポンスオブジェクトを丸ごとログ出力している**: 28行目 `logger.info("決済レスポンス: " + response)`。`response` の中に秘密情報が含まれていれば、それも一緒に漏れる。載せる項目は「秘密でないと確認したもの」だけに絞る。
4. **文字列連結でログを組み立てている**: `"..." + request.getCardNumber()` は、SLF4J のプレースホルダ `{}` を使うべき。無駄な文字列生成を避けられ、可読性も上がる。

### 【例外編】

5. **例外を握りつぶして `false` を返している（設計上の致命傷）**: 「カードが残高不足で拒否された（業務的失敗）」と「ネットワーク断でゲートウェイに繋がらなかった（システム故障）」が、どちらも同じ `false` になる。呼び出し側は失敗の理由を区別できず、リトライすべきか・ユーザーに何と伝えるかを判断できない。
6. **`e.printStackTrace()` を使っている**: 標準エラー出力（`System.err`）に垂れ流すだけで、ログ基盤に乗らない。時刻もレベルも付かず、本番では実質どこかへ消える。
7. **`logger.error(..., e.getMessage())` でスタックトレースを捨てている**: 一言メッセージだけを記録し、調査に一番必要な「どこで・どういう順で失敗したか」の足あと（スタックトレース）を失っている。第2引数に例外オブジェクト `e` そのものを渡せば、スタックトレースごとログ基盤に届く。

---

## ⭕️ 改善案 (good/ 配下)

1. **ログには秘密情報を載せない**: カード番号は下4桁だけマスク（`****1234`）、CVV・APIトークンは一切出さない。追跡には秘密でない `orderId` と `amount` を使う。
2. **プレースホルダ `{}` でログを組み立てる**: `log.info("... card=****{}", lastFour(...))` の形にする。
3. **レスポンスは必要な項目だけ出す**: 丸ごとではなく `success` / `resultCode` など安全な項目に限定。
4. **業務的失敗とシステム故障を分ける**: 「決済が成立したか」は戻り値 `boolean` で表し、システム的な失敗は独自例外 `PaymentGatewayException` を投げて呼び出し側に伝える。
5. **例外は「記録してから包んで投げ直す」**: `log.error("...", e)` でスタックトレースごと記録し（`printStackTrace()` は削除）、`throw new PaymentGatewayException(msg, e)` で原因例外を連鎖させて再送出する。
6. **（おまけ）コンストラクタインジェクション化**: `@Autowired` フィールド注入をやめ、`final` フィールド＋コンストラクタで依存を受け取る。テストしやすくなる。

## 💬 レビューコメント例文

> `charge()` のログ、調査用に丁寧に残してくれてありがとうございます！ここでひとつ一緒に直したい点があります。21〜24行目でカード番号・CVV・APIトークンがログに出ていますが、ログは運用や外部の監視基盤など色々な人の目に触れてファイルに残り続けるので、ここに秘密情報が載ると PCI-DSS 違反や情報漏洩に直結してしまいます。特に **CVV は保存・出力そのものが禁止**なので要注意です。追跡にはカード番号の下4桁マスクと `orderId` があれば十分なので、そこに絞りましょう。
>
> もう一点、`catch` して `false` を返している箇所について。今の作りだと「残高不足で拒否された」のと「通信が落ちて繋がらなかった」が同じ `false` になってしまい、呼び出し側が「リトライすべきか」「ユーザーに何と伝えるか」を判断できません。**通信エラー等のシステム的な失敗は例外として投げ直す**と、業務的な拒否（`false`）ときれいに切り分けられます。あわせて `printStackTrace()` は `log.error("...", e)` に置き換えると、スタックトレースごとログ基盤に残せて調査が楽になりますよ。
