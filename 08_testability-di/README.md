# 第8問：支払い金額を計算するサービス（テストのしやすさ）

**📅 出題日**: 2026年7月12日

## 📋 レビュー課題

ECサイトで、商品の「支払い金額」を計算する `PriceService` です。
ベース価格に対して、時間帯・タイムセール・会員ランクに応じた割増や割引を順番に適用し、
最終的な支払い金額を返します。

- 深夜(22時〜翌5時)なら 10% 割増
- タイムセール中なら 20% オフ
- GOLD 会員なら 100 円引き

ローカルで動かすと期待通りの金額が返ってきて、一見きちんと動いています。

対象: `bad/PriceService.java`

このコードを本番に出す前提で、「どこが問題か・どう直すべきか」をレビューしてみてください。
今回は特に **「このロジックをどうやって自動テストで守るか（テスト容易性）」** の観点で見てみましょう。

> ヒント：「深夜割増が正しく効くか」「セール中の金額が正しいか」「GOLD会員だけ100円引かれるか」を
> あなたが JUnit で確かめようとしたら、素直にテストが書けるでしょうか？書きにくいなら、それは**設計のにおい**です。

---

## ❌ NGポイント (bad/ 配下)

1. **時刻を自前で調達している（`LocalDateTime.now()` 直呼び）＝ 最重要**: メソッドの中で「今の時刻」を直接取得しているため、返る金額が**実行した時刻に依存**する。「深夜割増のテスト」を昼に流すと失敗し、深夜に流すと成功する——つまり**安定して自動テストできない**。時刻という“外の状況”をコード内部に固定で埋め込んでしまっている。
2. **依存を `new` でメソッド内生成している（`new SaleRepository()`）**: セール判定の窓口をメソッドの中で自分で生成しているため、テストから「今はセール中ということにする／しない」と**差し替えられない**。本物のDB等に密結合し、テストのたびに本物が動いてしまう（重い・不安定）。
3. **static ユーティリティへの隠れた依存（`MemberUtil.getCurrentRank()`）**: static 呼び出しは外からすり替えるのが非常に困難で、「この人はGOLD会員ということにする」というテスト用の状況が作れない。クラスのシグネチャ（コンストラクタ/引数）に依存が現れず、**何に依存しているかが隠れてしまう**点も保守上の問題。
4. **マジックナンバーの多用（10 / 5 / 100 / "GOLD"）**: 数字だけがコードに散らばり「この10は何％？」が読み手に伝わらない。仕様変更時に**修正漏れ・意味の取り違え**を招く。
5. **`System.out.println` によるログ出力**: 本番のログ基盤に乗らず、出力レベル（INFO/ERROR）の制御もオン/オフもできない。障害調査で役に立たない。

> 補足：レビューで挙がった「0除算の懸念」は、割る数が `10` `5` という**固定の定数**のため実際には発生しない（分母が変数で0になり得るときだけのリスク）。指摘は「どの行のどの分母が0になり得るか」を指差せるかで検証する癖をつけるとよい。

## ⭕️ 改善案 (good/ 配下)

1. **時刻は `Clock` を注入して受け取る**: `LocalDateTime.now()` をやめ、`Clock` をコンストラクタで受け取り `LocalTime.now(clock)` で取得。本番は実時刻の時計、テストは `Clock.fixed(...)` で「23時固定の時計」を渡せる → 深夜割増を安定して検証できる。
2. **依存はインターフェース化してコンストラクタで注入（DI）**: `SaleRepository` / `MemberService` をインターフェースにし、`new` せず外から差し込む。テストではラムダ式で偽物（`() -> true` など）を渡せる。
3. **static 呼び出しを廃し、依存をシグネチャに明示**: 何に依存しているかがコンストラクタを見れば分かるようになり、差し替えも容易になる。
4. **マジックナンバーを名前付き定数に**: `LATE_NIGHT_SURCHARGE_PCT = 10` のように意味を名前で表す。
5. **`Logger`（SLF4J）でログ出力**: レベル制御・プレースホルダ `{}` を活用。
6. **（副次効果）テストが書けるようになる**: 上記の結果、DBもサーバも起動せず `PriceServiceTest` 1ファイルで深夜割増・セール・GOLD割引を全て検証できる（`good/PriceServiceTest.java` 参照）。

## 💬 レビューコメント例文

> `calculatePrice` のロジック、仕様どおり動いていて良いと思います！ひとつ設計の相談で、ここを **あとから自動テストで守れる形**にしておけると安心かなと。いま `LocalDateTime.now()` / `new SaleRepository()` / `MemberUtil.getCurrentRank()` をメソッドの中で直接呼んでいるので、「深夜だったら」「セール中だったら」「GOLD会員だったら」という状況をテストから作れないんですよね。時刻は `Clock` を、セール・会員ランクはインターフェースにしてコンストラクタで受け取る形（DI）にすると、テストで自由に差し替えられて、金額計算が壊れた瞬間に気づけるようになります。あわせて 10 や 100 は名前付き定数に、`System.out.println` は `Logger` に寄せると読みやすさ・運用面も良くなりそうです。よければ一緒に直しましょう💪
